# Debug HUD

Оверлей для наблюдения за каждым этапом аудио/сетевого пайплайна в реальном
времени: здоровье RT-ядра PortAudio, пропускная способность IPC
(utility→renderer), статистика WebRTC-пиров, дрейф часов Web Audio, jank
главного потока, тренд кучи JS и (опционально) CPU/RAM по процессам. Сделан,
чтобы отвечать на вопрос «что сейчас работает, с какой скоростью, и что грузит
процессор, ничего при этом не делая».

Переключение — **Ctrl+Shift+D**. По умолчанию работает только в dev — в
production-сборке ничего не делает.

## Как перенести в другой клон репозитория

Вся папка `debug/` самодостаточна. Чтобы передать её человеку, работающему в
своей копии:

1. Скопировать `client/src/debug/` в его checkout по тому же пути.
2. Добавить одну строку в его `client/src/main.tsx` (после остальных импортов,
   место не важно):
   ```ts
   import './debug/install'
   ```
3. Запустить приложение как обычно (`npm run dev`). В dev-сборке HUD включается
   автоматически. Чтобы включить его в не-dev сборке, нужно выставить
   `VITE_DEBUG=1` в env, с которым собирается приложение.

Это всё — все вкладки, кроме **procs** и строк cadence opus/pcm во вкладке
**core**, заработают сразу, потому что они только читают API, которые
приложение уже предоставляет (`window.nativeAudio.getStats()`,
`RTCPeerConnection.getStats()`, `performance.memory`, `PerformanceObserver`).
Этим двум вкладкам нужно несколько дополнительных строк вне этой папки
(см. шаг 4) — без них они просто показывают «unavailable» и ничего не ломают.

## Опционально: CPU/RAM по процессам + cadence opus/pcm

Три файла вне `debug/` несут несколько строк проводки для двух метрик, которые
нельзя считать пассивно из рендерера:

- **`client/electron/main.js`** — добавляет
  `ipcMain.handle('debug:get-process-metrics', ...)` через
  `app.getAppMetrics()` (CPU% и RAM по процессам main/renderer/utility/GPU).
- **`client/electron/nativeAudio/preload.js`** — отдаёт этот хендлер как
  `window.kgbDebug.getProcessMetrics()` через отдельный вызов
  `contextBridge.exposeInMainWorld('kgbDebug', ...)`, специально не смешанный с
  `nativeAudio`, чтобы его отсутствие было безопасным.
- **`client/electron/nativeAudio/utilityHost.mjs`** — замеряет интервал между
  последовательными вызовами `onPcm`/`onOpus` (оба работают в одном JS-потоке
  utility-процесса, см. AUDIT.md §1.5) и добавляет четыре поля в существующий
  ответ `getStats`: `pcmIntervalMsAvg/Max`, `opusIntervalMsAvg/Max`.

Сравните эти три файла с версией в этом репозитории и повторите те же
небольшие изменения, если checkout партнёра успел разойтись. Ничего из этого
не меняет поведение в runtime — только добавляет read-only счётчики рядом с
уже существующим кодом (`getStats`, `getAppMetrics`).

## Заметки по архитектуре (почему сделано именно так)

- **Никакого патчинга методов на горячих путях.** Статистика RTC берётся из
  `RTCPeerConnection.getStats()` (браузер уже её считает); статистика IPC — из
  `nativeAudio.onPcm`/`onOpusPacket`, которые уже поддерживают несколько
  подписчиков через `Set` — этот коллектор просто ещё один подписчик, а не
  замена. Единственное, что реально патчится — глобальный конструктор
  `RTCPeerConnection` (чтобы регистрировать новые инстансы для опроса), и это
  происходит до того, как в приложении появляется хоть одно peer-соединение.
- **`window.nativeAudio` заморожен `contextBridge`.** Переназначение любого из
  его методов выбросит исключение. Поэтому `pushInboundOpus`/`pushSoftmix` не
  оборачиваются — входящий трафик измеряется через статистику
  `RTCDataChannel` вместо этого.
- **Дорогие коллекторы включаются только при видимости HUD.** Сэмплер FPS на
  rAF и быстрый опрос `procs` работают только когда HUD открыт
  (`debugBus.isHudVisible()`); всё остальное опрашивается в медленном фоновом
  режиме, чтобы инструмент не стал ещё одним «вечным RAF/poll-циклом», который
  AUDIT.md §9.C уже отмечает в других местах этой кодовой базы.
- **Метки времени из разных процессов никогда не сравниваются.**
  `timestampUs` у opus-пакетов берётся из часов utility-процесса; `performance.now()`
  рендерера — другой домен часов. Их сравнение дало бы число задержки, которое
  выглядит точным, но неверно. Там, где сравнение часов корректно (Web Audio
  `currentTime` против собственного `performance.now()` рендерера — один
  процесс), `webaudio.ts` это делает; везде остальное — только интервалы,
  измеренные в пределах одного домена часов.
