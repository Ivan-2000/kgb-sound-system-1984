Версия: **2.3**
Обновлён: 2026-06-06

> **Смена курса №2 (2026-06-06) — panels-first без кабелей.** Версия 2.x этого файла описывала кабели/функциональные связи между нодами как продуктовое ядро (G4 + G5 + G8). **Решение пользователя:** основное рабочее пространство — **Floating Panels**, каждая нода самодостаточна. Связей между нодами **нет** — за единственным исключением общего clock sync через проектный транспорт. Canvas остаётся **визуально** (расстановка нод + миникарта), но drag-to-connect и кабели **скрыты**, `ControlBus`-маршрутизация **выключена**. Код G4 не удаляется — замораживается в репозитории.
>
> **Что заменяет кабели:**
> - **Drum → Piano Roll** — разовый перенос паттерна (конвертация `drumState → NoteEvent[]`), результат становится миди-клипом на дорожке таймлайна. См. «Drum Machine» ниже и `TASKS.md` PR5.

> - **Звук миди-клипа** — переходный период: барабанные клипы играются голосами драм-кита; мелодические клипы — после VST.
> - **Аудио-маршрутизация** — DAW-цепочка `track → InsertChain (VST) → mixer master`; G5 (audio routing через граф) **удалён**.
> - **Math/util-ноды (G8)** — **удалены** из roadmap: требовали кабелей, без них продукт не зависит от них.
>
> **Смена курса №1 (2026-06-01).** Версия 1.0 описывала панели как фундамент, кабели/нодовую систему (C9) откладывала. В 2.x наоборот: нода-граф был объявлен фундаментом, а Panels и Canvas — двумя рендерами над ним. Часть архитектуры (G1–G3: ядро + синк + Panels) от той смены курса сохраняется и активно используется. ## Концепция

**Panels-first.** Floating Panels — основное рабочее пространство. Каждый модуль — самодостаточная нода с собственной панелью, обменивающаяся с системой только через общий проектный транспорт (clock sync). Между нодами **функциональных связей нет** — кроме исключений, явно перечисленных ниже.

**Canvas остаётся как визуальная альтернатива:** те же ноды на холсте React Flow, расстановка, миникарта, зум. Но кабели и drag-to-connect **скрыты**, `ControlBus`-маршрутизация **выключена**. Canvas — это «карта расстановки», не «модульный синт».

Состояние одно (граф нод как контейнер существования/параметров/позиций), видов два. Переключение вида не меняет граф.

### Принципы

- Каждый модуль — самостоятельная нода, регистрируется в `nodeRegistry`, вызывается через меню «+».
- **Сторонняя нода — JS/TS-модуль**, реализующий контракт (`NodeDefinition`). Контракт сохранён (типизированные порты, ParamDef), но рантайм-сигналы между нодами по портам **не маршрутизируются** в текущей версии. Песочница/подпись — позже (см. G6).
- **Граф общий на комнату** — синкается существование нод, параметры, позиции. Личное у каждого — режим Panels/Canvas и привязка к физическому устройству.
- **Транспорт** — единственный «общий сигнал»: один `Tone.Transport` на проект; Play/Stop/BPM в тулбаре, синкаются. Все ноды-секвенсоры (драм, будущие миди-клипы) подписаны на проектный транспорт и сами инкрементят свой шаг — это и есть весь clock sync.
- **Связи между нодами** — **отсутствуют**, за исключением:
  1. **Drum → миди-клип на таймлайне** — разовая операция переноса паттерна (ПКМ → «Перенести паттерн»). Не реалтайм-связь, а конвертация данных.
  2. **Миди-клип на таймлайне → инструмент дорожки** — стандартная DAW-цепочка `track → InsertChain → mixer`. Внутренняя кишка дорожки, не межнодная связь.
- **Аудио — нативно**. JS-граф не маршрутизирует звук; всё сводится в utility-процессе с PortAudio.
- Миксер открыт по умолчанию при входе в комнату.
- Транспорт (BPM / Play / Stop / метроном) — фиксированный toolbar, не нода.

### Замороженные подсистемы (код в репо, не используются)

- **Кабели Canvas (G4)** — `CableEdge.tsx`, `onConnect` в `CanvasView`, рендер `<Handle>` на портах. Скрыты UI-флагом / условным рендером. Не удаляются — могут пригодиться, если вернёмся к модульной модели.
- **ControlBus-маршрутизация** — `controlBus.emit/subscribe`/`setEdges` — код остаётся, но `setEdges` вызывается пустым массивом (или вообще не вызывается); `emit` не доставляет (нет route).
- **Прямой кабель piano→drum** — удаляется из активной логики через шаги PR1–PR5 в `TASKS.md` (Piano Roll становится редактором клипа).

## Контракт ноды

Это то, под что пишут сторонние ноды. Полная типизация — в [`client/src/graph/types.ts`](client/src/graph/types.ts).

```ts
PortKind     = 'audio' | 'control' | 'trigger' | 'value' | 'midi'
PortDef      = { id, label, kind, direction: 'in'|'out', multiple? }
ParamDef     = { id, label, type: 'number'|'enum'|'bool'|'string', default, min?, max?, step?, options? }
NodeManifest = {
  type, label, icon, description, version, author?, ports[], params[],
  singleton?,    // true → один экземпляр на комнату, id = type
  duplicable?,   // вместе с singleton → ПКМ «Дублировать» создаёт независимые копии
  local?,        // личная нода, не синкается (settings)
  defaults?      // panelPos / canvasPos / size — стартовые координаты
}

// Единственный интерфейс ноды к системе — она не знает про сеть, React и другие ноды:
NodeContext  = {
  nodeId
  emit(portId, payload)            // в выход → handler'ам подключённых входов (локально, через ControlBus)
  onInput(portId, handler)         // слушать вход → unsubscribe
  getParam(paramId) / setParam(paramId, value)   // setParam идёт через синк (общий)
  onParam(paramId, handler)        // реакция на изменение (локальное или удалённое)
  audio?                           // зарезервировано в типах; G5 (нативная маршрутизация через граф) удалён пивотом
}

NodeInstance = { render(): ReactNode, dispose() }     // нода несёт свой UI сама
NodeDefinition = { manifest, create(ctx): NodeInstance }
```

Автор экспортирует `defineNode({ manifest, create })`. Сигналы: `trigger` — момент (тик), `value` — длящееся значение.

> **⚠ Заморожено пивотом 2026-06-06.** `emit` / `onInput` / `ControlBus`-маршрутизация **в текущей версии не работают**: G4f-1..G4f-3 скрывают кабели и оставляют шину без рёбер. Контракт сохранён как поверхность типов, но рантайм-доставка между нодами отключена. Связи между нодами — только через специальные механизмы (перенос паттерна Drum→таймлайн, инструмент дорожки). Подробнее — раздел «Замороженные подсистемы» в Концепции.

### NoteEvent — payload на `midi`-портах *(добавлен в `types.ts` 2026-06-01)*

Музыкальные события несут данные (не просто «бан»), чтобы маски могли комбинировать по высоте/времени/velocity. Тип `NoteEvent` экспортируется из `client/src/graph/types.ts` (и из barrel `client/src/graph/index.ts`):

```ts
NoteEvent = { pitch: 0–127, velocity: 0–127, durationBeats?: number, id?: string }
```

- `midi`-порт несёт `NoteEvent`; `trigger` остаётся голым импульсом (клок, гейт) — не смешиваем.
- Голос интерпретирует `pitch`: мелодический (синт/питч-сэмплер) → частота; барабанный набор → карта `pitch → sample` (GM-style). Одна нота-валюта → маски/транспонирование/фильтры работают одинаково на мелодии и на барабанах.
- ~~`midi` — полноценный маршрутизируемый сигнал: любой `midi`-out → любой `midi`-in (как audio-кабель, только событиями).~~ **Заморожено пивотом 2026-06-06.** Структура `NoteEvent` остаётся валютой данных — используется при переносе паттерна Drum→миди-клип, в нотах пиано-ролла и в будущем при MIDI-инпуте с устройства. Но через `midi`-порты ноды друг с другом **не соединяются** — кабели/`ControlBus` отключены.

## Что общее, что личное

| Состояние | Область | Почему |
|---|---|---|
| Ноды (существование, тип, параметры) | **Общее** (синк) | Общий инструментарий комнаты |
| Связи (edges) | **Общее** (синк) | Активно не маршрутизируется (кабели заморожены), но синк-канал остаётся — на случай возврата |
| Позиции нод (panel + canvas) | **Общее** (синк) | Единое пространство: сдвинул — у всех сдвинулось |
| Цвет участника | **Общее** (синк) | Рандомный на участника при заходе; каналы/тайлы подкрашены; одинаков у всех для идентичности |
| Режим Panels / Canvas | **Личное** | Новичок смотрит панелями, профи — холстом; нельзя затащить насильно |
| Стэкинг (z-index), фокус | **Личное** | Каждый фокусирует своё |
| Привязка к физическому аудиоустройству | **Личное** | У каждого своё железо; «вход = ASIO id 16» существует только на его машине |
| **InsertChain (VST на канале/дорожке)** | **Общее по топологии, личное по бинарникам** | Список плагинов и параметров синкается; сам бинарник VST — у каждого свой, предупреждение если отсутствует |

Синк-мутации (Zod-события в `syncProtocol.ts`): `node_add`, `node_remove`, `edge_connect`, `edge_disconnect`, `param_change`, `node_move`, `node_resize`. Личные изменения (`setViewMode`, `focusNode`) идут мимо синк-воронки. `edge_connect/disconnect` сохранены, но кабели UI-уровня заморожены — события используются только если кабели вернутся.

## Ядро нод — спецификация

Активных нод в продукте — **четыре**: **Mixer**, **Timeline**, **Metronome**, **Drum Machine**. Все — граф-ноды (синкаются по механизму G2). Piano Roll после пивота **не самостоятельная нода**, а редактор миди-клипа на дорожке таймлайна (см. раздел 5). Math/util-ноды (раздел 6) удалены пивотом. Ноды контента строятся поверх инфраструктуры G1–G3 (G4 заморожен, G5/G8 удалены).

### 1. Mixer — аудио-хаб *(главная нода)*

> **Статус (2026-06-01):** реализовано и проверено в браузере. Вёрстка — горизонтальный ряд вертикальных стрипов (`.mixer-rack`). **Стрип сверху вниз:** имя, **вертикальный фейдер громкости** с бегунком + пик-метр, значение, кнопки **M / S / →**, **маленькая pan-крутилка** (влево/вправо), **круглая кнопка записи** в самом низу. Подкраска цветом участника. **Master** — то же самое (фейдер + pan + запись), **без слотов**. На участника — N стрипов (по каналу), все в его цвете. Запись (●) — armed-состояние (красный пульс); таймлайн уже подключён (см. раздел 2 Timeline), создание прокси-дорожки по Record работает. Реальный WAV-pipeline — Phase 3 T2. **Mic On / Cam On убраны из миксера → панель над видео-рамкой** (`.video-controls`). Общий компонент `MixerStrip` + `Knob` (крутилка = pan); `MixerChannel`/`RemoteChannelStrip`/`LocalMixerStrip` — тонкие обёртки. Цвета детерминированы от socketId (`participantColor`, без синка). Остаток: запись→таймлайн; pan реален только у web-audio каналов (`mixerEngine.setPan`), у native/local/master pan и local input gain — пока визуальные (нет поддержки в движке). *(Стрипы участников видны только в реальной сессии — в браузере соло виден лишь Master.)*

Сюда стекаются **все аудиопотоки всех участников**.

- Мастер-канал + каналы **создаются по мере захода участников**.
- От одного участника может идти **несколько потоков → отдельный канал на каждый поток** (модель M2/M3: `remoteChannelMeta` + `RemoteChannelStrip`).
- **Цвет участника** детерминирован от socketId; все его каналы подкрашены этим цветом.
- **Record на канале** → создаёт дорожку на Timeline и пишет туда (см. «Модель записи»).
- **InsertChain на канале (VST)** — компонент `<InsertChain target="channel" id={channelKey} />` в `LocalMixerStrip`/`RemoteChannelStrip`. На локальном input-канале цепочка применяется **до Opus-кодера** — собеседники слышат уже обработанный сигнал. На master — не применяется (на старте; решение по pre-master VST принимается позже).
- **MIDI-strip (отложено в конец roadmap, MI3–MI6)** — у канала появится тип `audio | midi`. MIDI-strip с обязательным instrument-slot (VST3-инструмент или встроенный синт-заглушка); MIDI → instrument → audio out канала → master. До MIDI-инпута с устройства этого нет.
- Порты: `audio` in (multiple), `audio` out (мастер). *(порты сохранены в манифесте на случай возврата к кабельной модели; для текущей логики не используются — звук не идёт через JS-граф)*
- Переиспользует `mixerEngine` + стрип-компоненты (M1–M5) — рестайл, не переписывание.

### 2. Timeline — монтаж *(Phase 3, новая, крупная; per-node экземпляр + дублирование готовы 2026-06-02)*

> **Per-node + дублирование (2026-06-02, проверено в браузере).** `useTimelineStore` (глобальный) заменён фабрикой `createTimelineStore()` (`counter`/`keyToTrack` теперь per-store); `TimelinePanel` принимает `store` пропом. Нода вынесена в `timeline/timelineNode.tsx` (`create()` делает свой стор, регистрирует в `timeline/timelineNodes.ts`, рендерит self-contained `TimelinePanel store={…}` через `getNodeInstance().render()` — `'timeline'` убран из App `panelContents`). Манифест `singleton:true` + `duplicable:true` (как драм): primary с id `'timeline'` (preload при входе, цель кнопки Record миксера через `getTimeline('timeline')`), ПКМ → Дублировать делает независимые копии. ### 4. Drum Machine — *(midi-in 2026-06-01; per-node экземпляр + per-node синк + дублирование готовы 2026-06-02)*

- **Per-node экземпляр — готово (2026-06-02), проверено в браузере.** Нода теперь владеет СВОИМ движком `DrumMachine` (создаётся в `create()`, регистрируется в `drumNodes.ts` по `nodeId`), а панель самодостаточна (`DrumNodeView` подписан на свой экземпляр) и рендерится через `getNodeInstance().render()` — `'drum-machine'` убран из App `panelContents` (как piano-roll).
